# v3.0

Release de **fundamentos a escala**: optimiza el backend, hace los planes gestionables,
añade rate limiting distribuido, caché por capas y SEO técnico. No introduce modos de evento
nuevos (eso es v3.1).

## Fase 1 — Performance backend

- `getEventStats()`: include anidado masivo → **3 queries paralelas**.
- N+1 de opciones de poll → **transacción** `deleteMany`/`createMany`.
- Importación CSV (participantes/categorías) → `createMany` en chunks de 50.
- Score de evento → `aggregate({ _sum })` en vez de cargar todos los votos.
- Dashboard: eliminada la 6ª query (colaboradores derivados de `_count`).
- `createEventParticipant`: 3 queries → 1.
- Admin/listings: `select` explícitos; `/polls` usa `groupBy` para el score.
- **Índices compuestos**: `Event(status,createdAt)`, `Vote(pollId,createdAt)`,
  `CollaboratorInvitation(eventId,status)`, `Report(isReviewed,createdAt)`.

## Fase 2 — Planes en base de datos

- Modelo `SubscriptionPlan` (límites/precio/stripePriceId/features).
- `getPlanFromUser` async desde BD con caché (`unstable_cache`, tag `subscription-plans`)
  y fallback fail-open.
- Split client/server: `plans.ts` (client-safe, fallback) vs `user-plan.ts` (BD).
- **Panel admin `/admin/plans`** (CRUD, solo ADMIN) con invalidación en vivo.

## Fase 3 — Rate limiting distribuido (Upstash)

- `src/lib/rate-limit-redis.ts`: sliding window, **fail-open**, headers RFC en 429.
- ~18 usos migrados en 16 rutas (mismos límites). Webhook Stripe exento.
- Eliminado el limitador in-memory.

## Fase 4 — Caché por capas

- `unstable_cache`: listado `/polls` (tag `events-public`, 60s), tags (300s), planes (1h).
- Invalidación con `revalidateTag` en mutaciones de evento.
- **Contadores Redis** `event:{id}:score|likes` (TTL 60s, fail-open).

## Fase 5 — SEO técnico

- `generateMetadata`/`metadata` en páginas públicas + canonicals.
- `noindex` para eventos privados/no aprobados.
- `sitemap.ts`, `robots.ts`, OG images dinámicas (`next/og`).
- JSON-LD `Organization` + `WebSite` (SearchAction) y `Event`.

## Fase 6 — Diseño

- `DESIGN.md` generado capturando el **estilo actual** de Pollnow (no rediseño): tema oscuro,
  acento azul/sky, bordes de 2px, motion con Framer Motion. Guía de consistencia para
  componentes nuevos.

## Fase 7 — Documentación

- README reducido; `/docu` con overview, setup, arquitectura, subsistemas, API, SEO,
  infraestructura y changelog. README v2.5 completo preservado en
  `docu/01-overview/full-reference.md`.
